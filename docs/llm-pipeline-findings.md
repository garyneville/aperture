## 2. Findings

### Finding 1: Groq rejects `json_schema` for `llama-3.3-70b-versatile`

**Severity:** Critical — blocks primary editorial path on every execution

**Error:**
```
400 - {"error":{"message":"This model does not support response format `json_schema`.
See supported models at https://console.groq.com/docs/structured-outputs#supported-models"}}
```

**Root cause:** The workflow sends:
```json
{
  "model": "llama-3.3-70b-versatile",
  "response_format": {
    "type": "json_schema",
    "json_schema": { "name": "…", "strict": true, "schema": {…} }
  }
}
```

Groq's structured outputs feature now only supports `json_schema` on these models:
- `openai/gpt-oss-20b` (strict + best-effort)
- `openai/gpt-oss-120b` (strict + best-effort)
- `openai/gpt-oss-safeguard-20b` (strict + best-effort)
- `meta-llama/llama-4-scout-17b-16e-instruct` (best-effort only)

`llama-3.3-70b-versatile` is **not listed**. It only supports the simpler `json_object` mode (no schema enforcement).

**History:** Execution 1995 (Apr 6) shows a different Groq error: `"Failed to validate JSON"` — suggesting Groq was previously attempting `json_schema` in best-effort mode on this model but removed support between April 6 and April 9.

**Workflow expression** (from `HTTP: Groq` node body):
```js
const structured = $json.editorialPromptMode === 'structured-output';
// structured=true → json_schema with strict:true (BROKEN)
// structured=false → json_object (would work)
```

The toggle exists but `editorialPromptMode` is set to `'structured-output'` in every observed execution.

### Finding 2: Gemini Inspire has never succeeded

**Severity:** High — inspire content completely absent from all briefs

**Error:** `429 — Request failed with status code 429`

**Root cause:** Timing analysis shows Gemini Inspire fires ~280ms after Gemini Fallback completes. The Gemini free tier's per-minute rate limit rejects this second call.

Execution 2058 timing:
```
HTTP: Gemini Fallback    start → +25,595ms (success)
  238ms gap
Code: Build Inspire Prompt     → +41ms
  0ms gap
HTTP: Gemini Inspire           → +94ms (429 rejected)
```

**Configuration gap:** The Inspire node has:
- `retryOnFail: false` — no retry at all
- No delay/spacing between the two Gemini calls
- No batching configuration

This means every single run makes two Gemini calls within ~300ms of each other, and the second one is always rejected.

### Finding 3: Gemini Fallback now also 429ing

**Severity:** High — fallback editorial path has degraded

**Pattern across executions:**

| Exec | Date | Mode | Groq | Gemini Fallback | Gemini Inspire |
|------|------|------|------|-----------------|----------------|
| 1995 | Apr 6 06:30 | trigger | ❌ 400 (json validate) | ✅ 200 | ❌ 429 |
| 2058 | Apr 9 06:30 | trigger | ❌ 400 (json_schema) | ✅ 200 | ❌ 429 |
| 2072 | Apr 9 21:41 | webhook | ❌ 400 (json_schema) | ✅ 200 | ❌ 429 |
| 2097 | Apr 11 06:30 | trigger | ❌ 400 (json_schema) | ❌ 429 | ❌ 429 |
| 2112 | Apr 12 06:30 | trigger | ❌ 400 (json_schema) | ❌ 429 | ❌ 429 |

Gemini Fallback worked on Apr 6–9 but started failing on Apr 11. Possible causes:
- Groq now returns its 400 error **instantly** (no wait), so the Gemini Fallback fires sooner relative to the previous workflow run's Gemini calls — potentially hitting the per-minute window
- Google may have tightened free-tier rate limits on the `gemini-3-flash-preview` model
- Other workflows on the same n8n instance (e.g. Gmail AI Email Manager, Recorder) may be consuming Gemini quota from the same API key

Gemini Fallback has retries configured (`maxTries: 2`, `waitBetweenTries: 3000ms`), but a 3-second backoff is too short for Gemini's rate-limit window.

## 3. Options

### Option A: Quick-fix Groq model swap

**Change:** Replace `llama-3.3-70b-versatile` with `meta-llama/llama-4-scout-17b-16e-instruct` in the `HTTP: Groq` node body.

**Scope:** Single expression change in `workflow/source/skeleton.json`

| Pros | Cons |
|------|------|
| Restores `json_schema` on Groq immediately | Llama 4 Scout 17B is smaller than 3.3 70B — may produce lower-quality editorial |
| No code changes needed | `strict: true` is not supported on this model (only `strict: false` / best-effort) |
| Free tier | Still depends on Groq availability |
| Keeps current app-layer contracts intact | Doesn't fix Gemini rate-limit issues |

**Effort:** Minimal — workflow-only change  
**Risk:** Low — worst case is lower editorial quality, still caught by existing validation

---

### Option B: Switch Groq to `json_object` mode

**Change:** Set `editorialPromptMode` to `'legacy'` (or modify the expression to always use `json_object`).

**Scope:** Config or expression change

| Pros | Cons |
|------|------|
| Works with the current model (`llama-3.3-70b-versatile`) | No schema enforcement — model may produce invalid JSON |
| Leverages the larger 70B model | Needs prompt to include JSON structure guidance more explicitly |
| No model migration risk | Existing `resolveEditorial()` validation handles this, but more retries likely |

**Effort:** Minimal  
**Risk:** Medium — may increase parse failures, but the validation + fallback logic already handles this

---

### Option C: OpenRouter migration (planned)

**Change:** Replace both Groq and Gemini HTTP nodes with OpenRouter calls, per `docs/openrouter-migration.md`.

**Scope:** Workflow nodes + 2 new adapter shims

| Pros | Cons |
|------|------|
| Single provider, single API key | Requires new adapters for response extraction |
| Free-tier models (`llama-3.3-70b-instruct:free`, `gemma-3-12b:free`) | More moving parts to change at once |
| Conditional fallback (call secondary only if primary fails) | Still free-tier — rate limits remain a concern |
| Already designed and documented | Needs testing before going live |
| Unified rate-limit handling | |

**Effort:** Medium — workflow + 2 adapter files + testing  
**Risk:** Medium — larger change but well-documented plan exists

---

### Option D: Direct Gemini API with structured output

**Change:** Replace Groq with Gemini as primary, using Gemini's native structured output (`responseMimeType: "application/json"`, `responseSchema`).

**Scope:** Workflow node + extractor changes

| Pros | Cons |
|------|------|
| Gemini 2.5 Flash has native JSON schema mode | Single provider = single point of failure |
| Eliminates Groq dependency entirely | Free tier rate limits still apply (and are already causing 429s) |
| Better model quality for creative writing | Buffer-body response format adds extraction complexity |
| Google's free tier is generous for low volume | Puts all eggs in one basket |

**Effort:** Medium  
**Risk:** High — rate limits are already the #1 problem with Gemini

---

### Option E: Hybrid — fix Groq + fix Gemini timing

**Change:** Combine quick fixes for both providers.

1. Swap Groq model to `llama-4-scout-17b-16e-instruct` (or switch to `json_object`)
2. Add Wait node (60s) between Gemini Fallback and Gemini Inspire
3. Increase Gemini Fallback retry backoff from 3s to 30s
4. Enable retries on Gemini Inspire (`maxTries: 3`, `waitBetweenTries: 30000`)

| Pros | Cons |
|------|------|
| Fixes all three failure modes | Increases total execution time by ~60–90s |
| No code changes — workflow-only | Doesn't address free-tier fragility long-term |
| Fastest path to working editorial | Still three separate providers to maintain |
| Preserves existing app contracts | |

**Effort:** Low — workflow node settings only  
**Risk:** Low

## 4. Recommendation

### Phase 1: Restore editorial today (Option E)

Make these workflow changes to `workflow/source/skeleton.json`:

1. **Groq model swap** — change body expression from:
   ```
   model: 'llama-3.3-70b-versatile'
   ```
   to:
   ```
   model: 'meta-llama/llama-4-scout-17b-16e-instruct'
   ```
   Also change `strict: true` to `strict: false` (this model only supports best-effort mode).

2. **Gemini Fallback backoff** — on the `HTTP: Gemini Fallback` node:
   - `maxTries`: 2 → **3**
   - `waitBetweenTries`: 3000 → **30000**

3. **Add Wait node** — insert a 60-second Wait node between `Code: Extract Gemini Fallback` output and `Code: Build Inspire Prompt` input.

4. **Gemini Inspire retries** — on the `HTTP: Gemini Inspire` node:
   - `retryOnFail`: false → **true**
   - `maxTries`: **3**
   - `waitBetweenTries`: **30000**

**Expected result:** Groq handles most requests. When Groq fails, Gemini Fallback retries with adequate backoff. Inspire fires 60s+ after Fallback, avoiding rate limits.

**Total execution time impact:** Worst case adds ~2min to a run that currently takes ~60s. The daily trigger at 06:30 has no time pressure.

### Phase 2: OpenRouter migration (this week / next week)

Follow the plan in `docs/openrouter-migration.md`:

1. Replace `HTTP: Groq` with OpenRouter primary (`llama-3.3-70b-instruct:free`)
2. Replace `HTTP: Gemini Fallback` with OpenRouter secondary (`gemma-3-12b:free`)
3. Add conditional fallback (only call secondary if primary fails)
4. Replace or disable Gemini Inspire
5. Add 2 adapter shims to keep app contracts stable
6. Keep `PHOTO_BRIEF_EDITORIAL_PRIMARY_PROVIDER=groq` for compatibility

**Benefits over Phase 1:**
- Single provider, single API key, single rate-limit pool
- Conditional fallback halves API calls in the happy path
- Eliminates Gemini buffer-body extraction complexity
- Path to paid tier if free models become unreliable

### Phase 3: Clean-up refactor (when stable)

1. Rename internal provider slots from `groq`/`gemini` to `primary`/`secondary`
2. Remove Gemini-specific buffer decoding from extractors
3. Add execution monitoring — alert when template fallback is used for >1 consecutive run
4. Consider whether the inspire branch adds enough value to justify the extra API call, or whether the spur-of-the-moment content should be generated as part of the main editorial prompt

## 5. Resolution (13 April 2026)

All three failure modes were resolved. The architecture was changed from sequential fallback to **parallel editorial calls**:

### Changes made

1. **Groq model swap**: `llama-3.3-70b-versatile` → `meta-llama/llama-4-scout-17b-16e-instruct` with `strict: false`
2. **Gemini model swap**: `gemini-3-flash-preview` (20 RPD) → `gemini-3.1-flash-lite-preview` (500 RPD)
3. **Parallel architecture**: Removed the conditional `If: Need Gemini Fallback` node. Both Groq and Gemini now fire in parallel after prompt construction. Gemini is delayed 30 seconds via `Wait: Gemini Editorial Delay` to avoid rate-limit collisions with Gemini Inspire.
4. **Gemini native structured output**: Upgraded `HTTP: Gemini Editorial` to use `systemInstruction`, `responseMimeType: 'application/json'`, and native `responseSchema` (with `additionalProperties` stripped recursively, as Gemini rejects it).
5. **Merge mode fix**: Changed `Merge: Editorial Route` from `append` to `combineByPosition` so both Groq and Gemini fields arrive in a single item to Format Messages.
6. **Editorial validation**: Added insight keywords (`despite`, `due to`, `although`, `however`, `prime`, `light pollution`, `has passed`, `passed`, `no further`, `narrow window`) to prevent false rejections.
7. **Gemini as primary**: `PHOTO_BRIEF_EDITORIAL_PRIMARY_PROVIDER=gemini` selects Gemini editorial when both pass validation.

### Current architecture

```
Schedule: 06:30 Daily
  → score weather, build prompt
    ├── HTTP: Groq (parallel)
    │     → Code: Inspect Groq Primary
    │       → Merge: Prompt + Groq
    │         → Merge: Editorial Route (input 0)
    │
    ├── Wait: Gemini Editorial Delay (30s)
    │     → HTTP: Gemini Editorial
    │       → Code: Extract Gemini Editorial
    │         → Merge: Prompt + Gemini
    │           → Merge: Editorial Route (input 1)
    │
    └── If: Run Inspire
          → Wait: Inspire Delay (30s)
            → HTTP: Gemini Inspire
```

`Merge: Editorial Route` combines by position → single item with all fields → `Merge: Prompt Context` → `Code: Format Messages` → app layer selects provider.
